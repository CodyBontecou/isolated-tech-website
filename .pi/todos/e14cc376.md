## Overview
Refactor the 8,500+ line `globals.css` into a modular, maintainable structure.

## Current State
- Single `app/globals.css` file with everything
- BEM-style naming conventions (good!)
- CSS custom properties for theming (good!)
- Hard to find/modify specific component styles

## Goals
- Improve developer experience finding styles
- Enable code splitting (only load needed CSS)
- Reduce cognitive load when editing
- Maintain existing visual design exactly

## Proposed Structure
```
app/
├── globals.css              # Imports only, minimal reset
├── styles/
│   ├── base/
│   │   ├── reset.css        # Box-sizing, margins
│   │   ├── typography.css   # Font families, sizes
│   │   └── variables.css    # CSS custom properties
│   ├── components/
│   │   ├── nav.css
│   │   ├── store-card.css
│   │   ├── store-hero.css
│   │   ├── app-page.css
│   │   ├── auth.css
│   │   ├── dashboard.css
│   │   ├── admin.css
│   │   ├── feedback.css
│   │   ├── roadmap.css
│   │   └── forms.css
│   ├── utilities/
│   │   └── helpers.css      # .sr-only, .dot, etc.
│   └── index.css            # Barrel file importing all
```

## Alternative: CSS Modules
Could migrate to CSS Modules for true component scoping:
- `StoreCard.module.css` alongside `StoreCard.tsx`
- Pros: True isolation, dead code elimination
- Cons: More files, lose global theming simplicity

## Recommendation
Start with file splitting (Phase 1), evaluate CSS Modules later (Phase 2).

## Success Criteria
- [ ] No visual regressions (screenshot comparison)
- [ ] Each component's styles in dedicated file
- [ ] Variables/theming centralized
- [ ] Build size same or smaller
- [ ] Developer can find any style in <30 seconds
