## Overview
Consolidate multiple database queries into fewer round-trips using JOINs, CTEs, and batch queries.

## Current State
Pages like `app/page.tsx` make 3-4 separate D1 queries:
1. Get featured app
2. Get all published apps
3. Get review stats for all apps
4. Get latest release dates

Each query is a network round-trip to D1.

## Goals
- Reduce D1 round-trips per page load
- Improve Time to First Byte (TTFB)
- Maintain code readability
- Keep query helper pattern from `lib/db.ts`

## Analysis Needed
1. Audit all pages for multiple sequential queries
2. Identify which can be combined vs must be separate
3. Benchmark current vs optimized queries

## Pages to Analyze
- `app/page.tsx` - Homepage (3-4 queries)
- `app/apps/[slug]/page.tsx` - App detail (7+ queries via Promise.all)
- `app/dashboard/page.tsx` - User dashboard
- `app/admin/*` - Admin pages
- `app/feedback/page.tsx` - Feedback list

## Optimization Strategies

### 1. CTEs (Common Table Expressions)
```sql
WITH app_stats AS (
  SELECT app_id, AVG(rating) as avg_rating, COUNT(*) as review_count
  FROM reviews WHERE is_approved = 1 GROUP BY app_id
),
latest_releases AS (
  SELECT app_id, MAX(released_at) as latest_release_at
  FROM app_updates GROUP BY app_id
)
SELECT a.*, s.avg_rating, s.review_count, r.latest_release_at
FROM apps a
LEFT JOIN app_stats s ON a.id = s.app_id
LEFT JOIN latest_releases r ON a.id = r.app_id
WHERE a.is_published = 1
```

### 2. D1 Batch API
```typescript
const [apps, stats, releases] = await env.DB.batch([
  env.DB.prepare("SELECT * FROM apps WHERE is_published = 1"),
  env.DB.prepare("SELECT app_id, AVG(rating)... FROM reviews..."),
  env.DB.prepare("SELECT app_id, MAX(released_at)... FROM app_updates...")
]);
```

### 3. Materialized Views (via scheduled worker)
Pre-compute expensive aggregations and store in a stats table.
