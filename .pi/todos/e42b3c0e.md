## Overview
Let happy customers share personalized discount codes and track referrals. Word-of-mouth is the best marketing for indie software.

## Goals
- Encourage organic sharing
- Reward existing customers for referrals
- Track referral effectiveness
- Build community/ambassador program

## How It Works

### For Referrers (Existing Customers)
1. Purchase an app
2. Get a unique referral code (e.g., `CODY20`)
3. Share code with friends
4. Get notified when someone uses it
5. (Optional) Earn credit/rewards

### For New Customers
1. Receive code from friend
2. Apply at checkout for X% off
3. Purchase completes
4. Referrer gets credit (if enabled)

## Database Schema

### New Table: `referral_codes`
```sql
CREATE TABLE referral_codes (
  id TEXT PRIMARY KEY,
  user_id TEXT NOT NULL,           -- Owner of the code
  code TEXT UNIQUE NOT NULL,       -- "CODY20"
  discount_percent INTEGER,        -- 20 = 20% off
  discount_cents INTEGER,          -- Or fixed amount
  times_used INTEGER DEFAULT 0,
  max_uses INTEGER,                -- NULL = unlimited
  is_active INTEGER DEFAULT 1,
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (user_id) REFERENCES user(id)
);
```

### New Table: `referrals`
```sql
CREATE TABLE referrals (
  id TEXT PRIMARY KEY,
  code_id TEXT NOT NULL,
  referrer_user_id TEXT NOT NULL,  -- Who shared the code
  referred_user_id TEXT NOT NULL,  -- Who used it
  purchase_id TEXT NOT NULL,       -- The resulting purchase
  discount_applied_cents INTEGER,  -- How much they saved
  referrer_credit_cents INTEGER,   -- Credit earned (if any)
  created_at TEXT DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY (code_id) REFERENCES referral_codes(id),
  FOREIGN KEY (referrer_user_id) REFERENCES user(id),
  FOREIGN KEY (referred_user_id) REFERENCES user(id),
  FOREIGN KEY (purchase_id) REFERENCES purchases(id)
);
```

### Extend `user` table (optional)
```sql
ALTER TABLE user ADD COLUMN referral_credit_cents INTEGER DEFAULT 0;
```

## Integration with Existing Discount System

You already have `discount_codes` table. Options:

### Option A: Separate System
Referral codes are distinct from discount codes.
- Pro: Clear separation, different rules
- Con: Duplicate code paths

### Option B: Extend Discount Codes
Add `referrer_user_id` to `discount_codes` table.
- Pro: Reuse existing checkout logic
- Con: Mixes admin codes with user codes

### Recommendation: Option A
Keep separate for cleaner logic and different business rules.

## Reward Models

### Model 1: Credit System
- Referrer earns $X credit per referral
- Credit applied to future purchases
- Never expires (or expires after 1 year)

### Model 2: Cash Out
- Referrer can request payout via PayPal/Stripe
- Minimum threshold ($25)
- More complex, tax implications

### Model 3: Recognition Only
- No monetary reward
- Just a thank you + leaderboard
- Simplest to implement

### Recommendation
Start with **Model 1** (Credit System) - gives value without cash complexity.

## Checkout Integration
```typescript
// In checkout flow
async function applyCode(code: string) {
  // First check discount_codes
  let discount = await getDiscountCode(code);
  
  // Then check referral_codes
  if (!discount) {
    const referral = await getReferralCode(code);
    if (referral && referral.user_id !== currentUserId) {
      discount = {
        type: 'referral',
        percent: referral.discount_percent,
        referrerId: referral.user_id,
        codeId: referral.id,
      };
    }
  }
  
  return discount;
}
```
